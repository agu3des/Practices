# Projetos de Sistemas em TI

**Introdução**

**Sistemas de Informação:** conjuntos de informações e procedimentos ordenados, utilizados para atender necessidades

- A + parte das empresas utiliza um sistema para controlar ou administrar
- Nesse contexto os dados são armazenados no sistema
- Além de armazenar as informações o sistema automatiza processos e guarda registros das operações feitas
- Para tais funções o sistema utiliza regras.

Um bom sistema opera o que lhe é necessário, mas não de forma precária. Um sistema trabalha em cima de: construção, implementação, avaliação e manutenção

**Metodologia:** Utilização de métodos, formas de trabalhos organizadas ou procedimentos para realizar uma tarefa. Desse modo, uma metodologia define os passos e as etapas para a realização de um trabalho.  No contexto dos sistemas de informação, as metodologias de desenvolvimento são formas de trabalho organizadas para planejar, construir, implantar, avaliar e manter sistemas de informação.

Três pontos devem ser considerados: quantidade de funcionalidades, cuidado com informações e preocupações com segurança, e facilidade de acesso.

Equipe de Profissionais: analistas, programadores, gerentes de projetos, outros

**ETAPAS:**

**1  - Planejamento:**

Elaboração de um desenho, daquilo que será feito, se faz levantamentos para estruturar o projeto do sistema.

- Entendimento do perfil e das necessidades do cliente: precisa conhecer quem irá utilizar o sistema quando ele estiver pronto. investigar expectativas e necessidade, descobrir como poderemos atendê-lo e identificar como e onde o sistema será utilizado
- Levantamento dos requisitos: requisitos são as descrições das funcionalidades do sistema, a partir das descrições os requisitos mostram como o sistema irá atender as expectativas do cliente.
- Identificação dos recursos e custos: tudo gira em questão do custo e do prazo

- Avaliação da viabilidade
- Elaboração do cronograma preliminar de trabalho: deve conter as principais atividades a serem realizadas, os responsáveis por cada uma delas e as datas de execução planejadas.

**2 - Construção**

É a parte do processo de desenvolvimento do sistema em que a programação é feita.

**3 ‒ Implantação**

É a disponibilização do sistema para o cliente. A implementação utiliza recursos locais - pc pessoal, nesse caso o trabalho do analista é necessário para entregar o sistema ao cliente. Já a implementação de sistemas mais complexos pode requerer a atuação de uma equipe com especialistas em redes, banco de dados e outros, desse modo pode envolver fornecedores de serviços de hospedagens ou de plataformas móveis.

Deve estar de acordo com o cronograma desenvolvido com o cliente

**4 ‒ Avaliação e Manutenção**

Avaliação de sistema em conjunto ao cliente. Sistemas mais complexos podem ter um contrato de manutenção. Neste contrato, são estabelecidas regras para suporte e eventuais atualizações.

**MODELOS**

**Modelo Cascata: Waterfall** - abordagem clássica, propõe uma abordagem sequencial para o desenvolvimento de um software, desde a comunicação inicial até a entrega.

1. Início do projeto, atividade mais relevante é o levantamento dos requisitos

[](https://lh7-us.googleusercontent.com/a1fIhleQlmgFoJegLXbXx3pghdGhyfelhlly8_0ahjEaFFZ3qW4vw6jeXFWgXZFUy4qEHEJtzk6qV-HLL8c7jbTRtVUGjGpeBtIXVdjU5KW1SsKhMZhPyyxK39y2bAFFi-y9WltBOi8LqIZSTPibwyg)

2. Estimativas de custos e prazos, além da elaboração do cronograma de desenvolvimento

3. Desenvolvimento do software, bem como a programação e a realização dos testes

4. O sistema é entregue e os usuários são preparados para usá-lo, quando entregue quer dizer que foi implementado

5. Oferecidos suporte técnico e manutenção aos usuários, Corretiva, para resolução de eventuais problemas. Evolutiva, para desenvolvimento de melhorias e novas funcionalidades.

**Modelo Incremental:**

é útil quando nem todas as informações para o desenvolvimento estão disponíveis, de modo a ser trabalhado em partes de forma que partes do sistema funcionem em tempo curto.

**Modelo Iterativo:** promove o processo de vários ciclos sucessivos ou iterações. Os ciclos aprofundam o conhecimento do sistema, tanto para o desenvolvedor quanto para o usuário final. Uma nova versão é construída a cada etapa, é útil quando existem mudanças ao longo do projeto.

[](https://lh7-us.googleusercontent.com/7k2GTpcvbGcVNdYxok0mHKE5GuObPbl72lkC2x4gs5BmiD3yU1_2CQLGmsRRbjSbeH5dYxlrd_z7A8ICUp6lsXeA8sYdK7QwxlOZvagREMNX0LmWIsp-LSbxp7U8KGZpNbFUYKxcGbKuOBDuYmNPbqM)

**Modelo de Prototipação:** também chamado de modelo evolucionário, por possibilitar o acompanhamento da evolução dos requisitos de projeto. A principal característica é a abordagem cíclica, por meio dos ciclos o objetivo é partir dos requisitos para a entrega de um protótipo rapidamente, a principal vantagem é a grande participação do cliente

Etapa 1 - Levantamentos: ocorre o entendimento das expectativas e das necessidades do cliente,bem como o levantamento dos requisitos

Etapa 2 - Modelagem rápida: modelagem –desenho da estrutura – do sistema de forma muito rápida. Nesse caso, não há preocupação com detalhes que possam tomar tempo, já que o objetivo não é construir algo que será mostrado ao cliente

Etapa 3 - Construção do protótipo:  a partir do desenho da fase anterior, é construído um protótipo para ser mostrado ao cliente. O protótipo pode ou não ser funcional.

Etapa 4 - Entrega e Avaliação:  o protótipo é apresentado ao cliente, que fará uma avaliação. A partir da avaliação, surgem ajustes nos requisitos e novas solicitações.

**Modelo Espiral:** o desenvolvimento de partes do projeto ocorre em paralelo, mas com a disponibilização rápida de partes funcionais para avaliação e testes. Uma característica que diferencia o Modelo Espiral dos demais é a utilização de uma estratégia baseada em ciclos, com a realização de uma análise de risco em cada ciclo. Isso reduz potenciais problemas e aumenta o grau de conclusão do sistema simultaneamente.

Como escolher o modelo?

1. Todas as informações estão disponíveis?
2. Há risco de mudanças nos requisitos durante o desenvolvimento?
3. É necessário mostrar alguma coisa pronta ao cliente em um curto espaço de tempo?
4. Existem pessoas experientes na equipe para trabalhar em várias frentes de desenvolvimento paralelo?

**Engenharia de Software**

→ Definição, especificação, desenvolvimento e a manutenção de sistemas de informação. Aplica metodologias de gestão de projetos, gestão de mudanças e outros.

→ Chegar ao produto final esperado é um ponto fundamental de todo o processo de planejamento e projeto de sistema. Caso as demandas e as expectativas dos futuros usuários do sistema não sejam bem-atendidas, não será possível conceber, projetar e desenvolver um sistema funcional.

Análise estruturada visa organizar o desenvolvimento do projeto e ampliar a comunicação com o usuário. Essa análise é composta de seis faces:

1. Concepção do software:
- Quais informações serão disponibilizadas na saída?
- Quais informações serão necessárias na entrada?
- Como o sistema será acessado?
- Quais benefícios trará para seus usuários(organização ou pessoas físicas)?
- Quais funcionalidades terá?
- Para que o sistema será desenvolvido?
1. Estudo de viabilidade
- Algumas questões têm de ser necessariamente verificadas. Entre outras coisas, é preciso analisar a disponibilidade de recursos – financeiros, técnicos e humanos – para o desenvolvimento do sistema. A ideia pode ser ótima, mas o custo e o prazo de desenvolvimento podem ser altos demais.
- A análise de viabilidade é feita com maiores detalhes após os levantamentos de requisitos e deve ser discutida entre desenvolvedores e clientes. Isso assegura que todas as questões técnicas, operacionais e econômicas sejam vistas. Dessa maneira, a decisão de prosseguir ou não com o desenvolvimento será acertada.
1. Projeto lógico
- Desenho das atividades que serão cobertas pelo sistema. Feito em formato de fluxograma ou diagrama de fluxo. Nesse caso, o diagrama de fluxo é uma representação gráfica do passo a passo dos processos que serão automatizados pelo sistema ou integrados a ele.
1. Projeto físico
- São definidas as linguagens utilizadas na programação, nos bancos de dados, nos ambientes de desenvolvimento e testes, e nos emuladores. Além disso, são determinados os computadores e os demais dispositivos requeridos para desenvolver, testar, implantar etc.
1. Implementação
- Momento em que o sistema será disponibilizado para os usuários ‒ seja no ambiente da organização, seja nos dispositivos dos consumidores finais.
1. Manutenção
- O desenvolvedor precisa prever atualizações em seu sistema. Tais atualizações devem trazer novas funcionalidades, garantir a compatibilidade com novos sistemas operacionais e equipamentos, ou corrigir erros.

Levantamento de requisitos: requisitos são funcionalidades solicitadas pelas pessoas que encomendam o sistema, tais pessoas são chamadas de requisitantes. Muitas vezes as pessoas que solicitam as funcionalidades têm uma visão do que necessitam.

1. Busque entender a finalidade de um sistema muito claramente, cada sistema tem uma razão de existir
2. Desenvolva com simplicidade, pois sistemas são difíceis de manter e de sofrer evoluções.
3. Visão clara do que o cliente pediu, não deve-se criar funcionalidades a mais nem esquecer o que foi pedido
4. Esteja pronto para evolução

Requisitos Funcionais: correspondem a tudo que o sistema deverá fazer, conjuntos de funções do sistema

Requisitos não Funcionais: relativos ao uso do sistema, ao desempenho, a segurança e a disponibilidade

Técnicas de levantamento de requisitos: Quando acontecem divergências ou conflitos, é preciso esclarecer e alinhar os requisitos com os gestores do projeto. Algumas técnicas que ajudam no levantamento de requisitos: entrevista (mais tradicional, conversar com os requisitantes sobre suas demandas), workshop (O workshop é uma variação da técnica de entrevista, já que as questões levantadas são as mesmas. A diferença é que o workshop é aplicado a um grupo de pessoas. Desse modo, espera-se que a resposta de uns estimule a opinião dos demais), questionário (utilizados como apoio a entrevistas ou enviados a pessoas que não podem ser entrevistadas pessoalmente, questões simples e diretas), brainstorming (é uma tradicional técnica para geração e captação de ideias. Desse modo, o brainstorming pode ser muito útil quando percebemos a dificuldade de transformar os desejos dos requisitantes em requisitos.), observação (bastante útil quando o sistema a ser desenvolvido vai automatizar ou apoiar um processo de trabalho já existente. Nesse sentido, os analistas podem entender o que o sistema deverá fazer a partir da observação do fluxo de trabalho.) e prototipação (Um protótipo pode ser utilizado para aprofundar os requisitos e avaliar alternativas da interface com os usuários).

Classificação dos Requisitos: ordem de importância e solicitações dos requisitantes.

- Desejáveis: não são necessários para o sistema estar completo
- Importantes: não impedem o sistema de funcionar, mas devem ser atendidos o quanto antes
- Essenciais: são fundamentais para o funcionamento do sistema

**Modelagem e Arquitetura**

A modelagem do sistema e de seus dados consiste na criação de um modelo que reflita o funcionamento e o comportamento de um software. Os modelos permitem demonstrar como serão montadas as estruturas do sistema e de seus dados, bem como seu relacionamento.

A arquitetura do sistema define as estruturas lógica e física para assegurar desempenho e confiabilidade ao sistema.

Além da visão estrutural, a concepção da arquitetura de um sistema deve considerar as perspectivas contextual (A perspectiva contextual considera o ambiente do sistema e suas conexões externas) e comportamental (A perspectiva comportamental considera como o sistema se comporta).

A modelagem comportamental consiste de alguns elementos. Entre esses elementos, temos:

Diagrama de Fluxo de Dados (DFD) - representação gráfica de como as informações irão fluir no sistema, é usado para organizar o sistema a ser feito, definir entradas, saídas e processos,

processos - círculos, agentes - retângulos, repositórios de dados -  linhas, fluxo de informações - setas
